# Knowledge Base Tiering Architecture

Luminatick introduces a new two-tier Knowledge Base architecture to safely manage both public-facing support content and internal operating procedures within the same ecosystem.

## Two-Tier System: Answers vs. SOPs

The Knowledge Base categorizes articles into two distinct tiers (`tier` column in the database):

1. **Answers (`answer` tier):**
   - Public-facing, user-friendly articles designed to be consumed directly by customers.
   - Contains general troubleshooting steps, FAQs, and product guides.
   - Used by the AI to automatically draft responses to customer inquiries, tickets, and widget chats.

2. **SOPs (`sop` tier):**
   - Standard Operating Procedures meant strictly for internal agent use.
   - Contains internal workflows, escalation policies, sensitive instructions, and agent guidelines.
   - Explicitly hidden from customers and the customer portal.

## Vectorize Fallback Search

To ensure agents can efficiently find relevant information across both tiers, Luminatick employs a robust search architecture utilizing Cloudflare Vectorize for semantic understanding:

- **Primary Search:** When searching for knowledge, the system can query the database directly for metadata matches (e.g., categories, direct text search).
- **Semantic Fallback:** If primary filters do not yield strong exact matches—or to enrich search results with conceptual relevance—the system performs a fallback search using Cloudflare Vectorize.
- **How It Works:** Vectorize compares the semantic meaning of the query (using embeddings generated by Cloudflare Workers AI) against chunked vectors of the knowledge base articles. This surfaces conceptually relevant information across both Answers and SOPs, even if exact keywords aren't present in the query.

## Dynamic AI Suggestions & Preventing SOP Leaks

Security and data privacy are paramount. To guarantee that internal SOPs are never leaked to customers through AI-generated responses, the system employs strict, dynamic context filtering at the database and vector query level:

- **Customer Inquiries (AI Auto-Drafts & Public Chat):** When the AI (e.g., Llama 3 / DeepSeek) generates a draft response for a customer or interacts via the public web widget, the retrieval context window is strictly filtered to **only include vectors/articles classified as the `answer` tier**. SOPs are systematically excluded from the Vectorize query parameters and database lookups. This ensures the AI never "sees" internal procedures when speaking to or drafting for a customer.
- **Agent Assistance (Internal Context):** When an authenticated agent explicitly interacts with the system or requests AI assistance for internal guidance, the AI's retrieval context is dynamically expanded based on their role. In this scenario, both Answers and SOPs are included in the search and context window, allowing the AI to guide the agent using internal policies without exposing them externally.

This robust filtering happens at the data retrieval layer, before any prompt reaches the Large Language Model (LLM). This architectural decision guarantees that sensitive internal procedures cannot be extracted by malicious actors through prompt injection or standard customer inquiries.

## Recent Architecture Optimizations & Security Mitigations

To further refine the AI suggestion feature and adapt to infrastructure changes, several key enhancements have been implemented:

### 1. Unified Vector Search & Threshold Adjustments
Previously, the system performed sequential vector queries for the `answer` and `sop` tiers. This has been refactored into a **unified vector search** that queries both tiers simultaneously. To support this:
- **Similarity Thresholds & Top-K Limits:** The vector search `topK` parameter has been significantly increased, and similarity score thresholds have been finely tuned. This guarantees that highly relevant SOPs are not prematurely pushed out of the retrieval window by generic, higher-scoring `answer` tier articles.

### 2. R2 Body Hydration for AI Context
In response to the "Option 3" hybrid storage migration—which offloaded large article bodies from D1 to Cloudflare R2 to bypass SQLite limits—the AI context retrieval pipeline was updated. The system now seamlessly performs **R2 body hydration**. After retrieving the relevant vector metadata from D1, the agent fetches the full article payloads via pre-signed R2 URLs or direct bucket bindings, dynamically assembling the necessary context window for the LLM.

### 3. Prompt Injection Defense & DoS Mitigations
With expanded context windows, new security boundaries were enforced:
- **Prompt Injection:** System prompts have been fortified with explicit boundary markers and adversarial instruction safeguards, ensuring user-supplied ticket text cannot override the core AI directives.
- **DoS Memory/CPU Limits:** Explicit safeguards limit the maximum aggregated token length and strictly bound the context window size before passing it to the Cloudflare Workers AI. This prevents malicious actors from triggering out-of-memory (OOM) errors or excessive CPU execution times through massive, concatenated payloads.
